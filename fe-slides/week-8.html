Static Site Generators

In the past years we have seen a come-back of static sites. Let's explore the differences between a static and a dynamic site.

Static sites

  • A site made up of HTML, CSS and JavaScript.
  • Each page is created and stored in the server and displays the content as is.

Dynamic sites

  • A site created using server side language, allowing for the content to be displayed dynamically, or on-the-fly.
  • The content is stored separately from the layout, and the page is dynamically created upon requested.
https://chandulanethmal.github.io/programming/web/Static-Dynamic-Websites/

Pros and Cons of Static and Dynamic Sites

Static Sites

+

  • Faster Loading: Nodatabase, no client server infrastructure.
  • More Secure: Less places to attack from.
  • Hosting is cheaper.

-

  • Limited Scalability.
  • Harder to manage.

Dynamic Sites

+

  • Easy to update and maintain.
  • Content can be tailored to the user.
  • Increased functionality

-

  • It takes more resources to create.
  • It can slow down due to performance uses.
  • More vulnerable to attacks.

Static Sites Generators

  • A static site generator is a tool that generates a full static HTML pages based on raw data and a set of templates simplifying the task of coding individual HTML pages.
  • Websites generated by a SSG have the speed and security of a static website and some of the advantages of dynamic sites.
  • Because static site generators create webpages in advance, webpages load slightly faster in users' browsers.
  • In other words, the static site generator takes content and turns it into flat HTML files ready to de displayed.
  • They are very lightweight.
  • SSG are a step between static and dynamic sites.
https://www.netlify.com/blog/2020/04/14/what-is-a-static-site-generator-and-3-ways-to-find-the-best-one/

Jekyll

  • Jekyll is a static site generator created in the Ruby language and distributed by Github.
  • It was created as a blogging platform.
  • It integrates with Github Pages which provides a free hosting option. (for a personal or a project site. Not a commercial one).
  • You can install Jekyll locally or directly on GitHub.
  • If Installing locally, you need to install / update Ruby.

Working with Jekyll

Things you need to know:

  • You can use HTML to create your content.
  • You can also use Markdown, which is a light markdown language used in plaintext files.
  • Markdown files will be converted to HTML.
  • The template language is Liquid.
  • Front Matter is metadata you add on top of the page that tells Jekyll how to parse the page.

Setting up your first site

  1. Create a New repository on Github.
  2. Add a Readme.md file
  3. Go to Settings/Pages
  4. On the Theme Chooser Select a theme.

What is Markdown?

  • Created in 2004, MD is a lightweight markup language to format elements in plain text documents.
  • Markdown can be used for anything. Websites, documents, email messages etc.
  • Is portable. Markdown files can be opened using any application.
  • MD is platform independent, so it runs in any Operating system.
  • MD is used by many applications. Reddit and Github support it.
  • Markdown files have an extension .md or .markdown
  • Text written in MD is converted into HTML that is capable of "parse" or process the md file.

Jekyll File Structure

_config.yml

  • This is the configuration file.
  • Used to set variables used on the site.

_posts

  • Where you create the blog posts if you are creating a blog site.

_includes

  • Used to store reusable snippets of code that can be displayed anywhere on the site.
  • Jekyll refers to these files as partials.

_site

  • Holds the generated HTML files of the site.
  • In Github local projects it might not be present.

_layouts

    This is where you store the template layouts or override an external layout.

_sass

  • This is where you will store all the syle files. Sass and css.

Jekyll Pages

  • Pages are created with the extension .md
  • You can use either HTML or Markdown, and they will be converted into HTML.
  • index.md, about.md are pages ususally created by Jekyll, but you can add as many as you need, following the same structure you would follow in any website.

Understanding the Jekyll structure.

  • After installing the basic Jekyll template on Github you will notice 2 things:
    1. It creates a gh-pages branch automatically. This is where the template will be installed and the one you should select when publishing your site.
    2. It only installs 2 files: _config.yml and index.md

Where are the template files?

  • All templates available on Github are available on https://github.com/pages-themes.
  • You can modify and override template settings by copying the files you want to modify from the template files to your site.

Configuring Jekyll on the _config.yml file

  • In this file we can include the configuration for the site by creating variables that can be accessed on any page.
  • Title. title: Page Title This is the site title, but is used when there is no page title available.
  • Description. description: Page Description. Used when no page description is available.
  • Theme Name. theme: jekyll-theme-cayman. The name of the theme you;re using.
  • Site URL. url: https://yourusername.github.io. This is the full url of your site.
  • Base URL: baseurl: /jk-assignmemt. Add this if you are using a Github site on a subfolder or repository.
  • Custom variables. There can be any type of content.
    instagram: "https:instagram.com/myaccount"
    you can use this variable later in any page referring to it as:
    [instagram]({{site.instagram}})

Jekyll Variables

Jekyll variables are used by wrapping them in {{ }}. There are some that are created by default and some that you can modify. You can used them on any template.

There are three main types of site variables in Jekyll:

Global Variables

  • They define global setting for your site. Like site and layout settings.

Site Variables

  • They are available to all pages in the site and also define elements that can be used on any page. {{site.tile}}

Page Variables

  • They are available at the page level and can be defined on the Front Matter section. {{page.title}}

Front Matter

Front Matter stores information about the page. Title, author, etc.

  • The front matter must be the first thing in the file and must take the form of valid YAML set between triple-dashed lines. ---
  • You can add pre-defined or custom variables and access them with template tags.
  • Front Matter created on the yml file will apply to all pages unless you override it on the page level.

Adding custom CSS

Jekyll uses SASS (Syntactically Awesome Style Sheets) by default.
SASS is and extension of CSS that enables you to use variables, nested rules, imports and more. But you can also use plain CSS.

How to add Custom CSS

  1. If you don't have it already. Create an assets folder. This folder is used for any file included on your site that does not have to be processed by Jekyll and will be used "as-is".
  2. Create a css folder inside.
  3. Create a style.css file.
  4. Import the site theme by adding this to your file:
  5. ---
    ---
    @import "{{ site.theme }}";

You can override classes and use your coustom classes after the @importdeclaration.

If you want to override the template css variables (located on the theme's sass folder), you have to do it right before the @import declaration.

Adding non-Jekyll files

Any file that does not have Front Matter is not processed by Jekyll.

These files include: images, .css, .js, and any other file that you want to include.

You need to keep these files in a folder. It is recommended to use an assets folder and add all the files there.

Note that the folder does not start with an underscore. That indicates Jekyll that the files inside are not Jekyll files.

An Introduction to Liquid

Liquid is a templating language that allows you to use variables, loops, conditionals to create templates.

  • Liquid has 3 types of markup:
    • Objects: which tell Liquid to output predefined variables as content on a page by enclosing them in {{ page.title }}.
    • Tags: which can contain logic and is surrounded by {% include disclaimer.html %}
    • Filters: change the output of a Liquid object. They are used within an output and are separated by a | {{ "hi" | capitalize }}. A list of available filters can be found here.
    • Comment: Any content that you put between {% comment %} and {% endcomment %} tags is turned into a comment.

You use the liquid markup in combination with the Jekyll Variables.

Layouts

Layouts are templates that can be used by any page, and wrap around page content. They are stored in a directory called _layouts

You can have more than one layout for different page types.

Layouts use a combination of Markdown and html, mixed with Liquid tags and Jekyll variables.

By using layouts you can share one element with many pages. For example, the element can be created once and shared across all templates.

Includes

Includes are reusable snippets of code that can be displayed anywhere on the site.

These files are .html files stored in the _includes folder.

Files can be included in templates or pages using the following code:
{% include file.html %}

Loops and Conditionals

Liquid tags can be used to add logic to the templates.

Loops

{% for item in array %}
{{ item }}
{% endfor %}

Conditionals

{% if user.name == 'tobi' %}
Hello tobi
{% elsif user.name == 'bob' %}
Hello bob
{% endif %}

Filters

Filters modify the result of the output statement. You can invoke filters by following the output statement's main expression with a |

Hello {{ 'tobi' | upcase }}

Common filters are:

  • append
  • capitalize
  • downcase
  • prepend

A list of filters is available here.

Themes

Jekyll themes set default data, layouts, includes, and stylesheets. However, you can override any of the theme defaults with your own site content.

Not all themes are compatible with Github pages, so make sure to select a non gem-based theme

Jekyll will look first to your site’s content before looking to the theme’s defaults for any requested file in the following folders:

  • /assets
  • /_data
  • /_layouts
  • /_includes
  • /_sass
